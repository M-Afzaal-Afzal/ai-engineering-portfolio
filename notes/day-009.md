# Day 9 Notes - Create Ticket Flow

## Goal

Add a professional create-ticket experience to the existing SupportDesk AI
dashboard without connecting to the backend yet. Prove the frontend workflow
with local state, client-side validation, inline errors, and success feedback.

## What Was Built

1. Added a `New Ticket` action near the main dashboard header.
2. Built a right-side `CreateTicketPanel` form for:
   - Subject
   - Customer email
   - Priority: Low, Medium, High
   - Description
3. Added manual client-side validation with typed form values and validation
   errors.
4. Added inline field errors and `aria-invalid` state for invalid inputs.
5. Created local ticket objects on valid submit with no backend API call.
6. Inserted newly created local tickets at the top of the visible ticket table.
7. Added success feedback after a valid local submission.
8. Kept existing sample tickets as the initial local dashboard state.
9. Refactored ticket types, seed data, and local ticket helpers into focused
   files so `page.tsx` stays easier to maintain.

## Key Concepts Learned

1. Local frontend workflows can validate product behavior before API integration.
2. Manual validation is a good fit for small forms when it avoids unnecessary
   dependencies.
3. Accessible forms need labels, semantic structure, inline error messages, and
   invalid state that assistive technologies can detect.
4. A right-side panel works well for dashboard creation flows because the table
   context remains visible behind the workflow.
5. React state should be updated from event handlers rather than effect-based
   synchronization when the change is caused by user interaction.
6. Shared TypeScript types keep page, component, and helper boundaries aligned.

## Short Reflection

Day 9 turned the dashboard from a static support view into an interactive local
workflow. The most important decision was keeping the form validation manual:
four fields do not need `zod`, `react-hook-form`, or resolver packages yet. The
clean-code pass also mattered because `page.tsx` was starting to absorb too many
responsibilities. Moving ticket data, shared types, and local ticket helpers out
of the page keeps the feature easier to extend when the backend connection is
added later.
