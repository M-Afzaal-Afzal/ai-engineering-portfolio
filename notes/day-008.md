# Day 8 Notes - Next.js App Shell

## Goal

Build the Week 2 Next.js frontend foundation: a polished Support Ticket Dashboard
shell with navigation, reusable UI components, and a static dashboard layout.
No backend integration yet. Proves full-stack fundamentals before adding AI features.

## What Was Built

1. **Next.js app shell** inside `apps/web` (already bootstrapped, now populated)
2. **Reusable UI components** (`Button`, `Input`, `Card` family)
3. **Sidebar navigation** with `SupportDesk AI` branding, Day 8 badge, four nav
   items (Overview, Tickets, Customers, Settings), and active-route highlighting
4. **Dashboard page** replacing the default Next.js home page:
   - Page title and description
   - Three metric cards: Open Tickets (24), High Priority (7), Avg Response (4.2h)
   - Static recent tickets table with five realistic rows, status and priority badges
   - Filter panel with keyword search Input, status/priority toggle buttons, and
     Apply/Clear Buttons
   - Week 2 Milestone card listing shipped and upcoming features
5. **Design system** using OKLCH color tokens registered via Tailwind v4 `@theme`
6. **Responsive layout**: dark sidebar on desktop (`lg:flex`), top nav bar on mobile
7. **App metadata** updated to `SupportDesk AI` with a portfolio description

## Key Concepts Learned

1. **Next.js 16 App Router**: file-system routing, `layout.tsx` as the root shell,
   `page.tsx` as the route content, server components by default.
2. **Server vs Client Components**: sidebar uses `usePathname()` for active state
   so it needs `"use client"`. Pages and static UI components stay as server
   components to reduce the client bundle.
3. **Tailwind CSS v4**: CSS-first configuration using `@theme inline` to define
   design tokens instead of a `tailwind.config.js`. Token names like
   `--color-accent` automatically generate `bg-accent`, `text-accent`, etc.
4. **OKLCH color model**: perceptually uniform color space. Reduces chroma near
   lightness extremes to avoid garish output. All palette values tinted toward
   the brand hue (hue 264, blue-slate family).
5. **Component composition pattern**: `Card` is split into `Card`, `CardHeader`,
   `CardTitle`, `CardContent`, `CardFooter` to allow flexible assembly without
   nested cards or prop-explosion.
6. **TypeScript discriminated union types** for `TicketStatus` and `TicketPriority`
   with record-type style lookup tables for badge colors.
7. **Accessible HTML**: semantic `<main>`, `<aside>`, `<header>`, `<nav>`,
   `<table>` with `aria-label`, `aria-current="page"` on active nav links.
8. **`@typescript-eslint/no-empty-object-type`**: empty interface extending a
   supertype should be rewritten as a type alias.

## Reflection

Day 8 closes the gap between the backend-only Week 1 and the full-stack AI
product Week 2 is building toward. The dashboard shell is clean, recruiter-
reviewable, and deliberately static: no API calls, no auth, no database.
That restraint is the point. Every piece of the UI is wired to a real
upcoming milestone, and the component structure is ready to absorb API
integration, authentication, and AI triage features without a rewrite.
The biggest design lesson was applying OKLCH color tokens via Tailwind v4
`@theme inline` instead of reaching for a `tailwind.config.js` — the CSS-
first approach is cleaner and aligns with where the ecosystem is heading.
